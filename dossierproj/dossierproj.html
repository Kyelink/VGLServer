Introduction

Projet : Application mobile VGL (Video Games Library)

Contexte : Projet de fin de formation CDA à l'école M2I

Durée : 22 mois

Technologies : React Native/Expo, Node.js/Express, MongoDB, Render

Objectif : Développer une application mobile multi-couche servant d'annuaire de jeux vidéo connecté à une API REST, avec système d'authentification et de recommandations personnalisées (vision d'évolution vers une plateforme e-sport) qui prend en compte le respect des RGPD, l'accessibilité et l'éco-conception.

Problématiques

  • Comment créer une application mobile performante et évolutive permettant aux utilisateurs de découvrir, organiser et avoir des recommandations de jeux vidéo ?
  • Comment automatiser la récupération de données obtenues sur le web via webscraping (dans le cadre légal d'un projet scolaire, non commercial) ?
  • Comment gérer les problèmes liés à la variation de certaines données en temps réel (nombre de likes d'un jeu qui évoluent constamment et casseraient une pagination en modifiant un ensemble ordonné selon les likes) ?

L'application a pour public cible les utilisateurs mobiles, Android dans un premier temps, mais avec possibilité d'évolution sur le marché IOS dans un second temps, puis à plus long terme, sur le web, c'est pourquoi j'ai choisi React Native comme technologie principale pour mon application. Le point fort de React Native est la portabilité d'une plateforme mobile à l'autre, en revanche, il souffre d'une moins grande portabilité vers le web, pleinement contrebalancé par la facilité de migration d'un projet React Native vers un projet React.js qui, lui, sera parfaitement adapté à une évolution future sur du web.

Le développement sur Expo permet d'utiliser son mobile en tant qu'environnement de développement, ce qui facilite et accélère énormément la boucle "modification du code"/"observation des modifications", et permet d'être au plus proche des conditions d'utilisation prévues de l'application, ce qui est le point faible du développement mobile sur émulateur, qui est obligé de simuler logiciellement toutes les interactions utilisateur matérielles, que l'on ne peut faire que sur mobile.

Le développement sur émulateurs, via Android Studio, a quand même été retenu comme solution viable, pour tester la responsivité de l'application à différentes tailles d'écran, ce que ne permet pas l'utilisation de son propre mobile.

Personnellement, le seul autre point faible que j'ai trouvé à l'utilisation d'Expo pour développer efficacement, est son opacité sur ce qui se passe en partie "réseau", c'est pourquoi j'ai pris l'habitude de l'utiliser avec sa dépendance web, qui facilite énormément le débug en observant ce qui se passe sur la partie "network" du navigateur qui exécutera notre application.

Dans la première version de l'application, je travaillais avec des collaborateurs, nous nous communiquions nos ajouts dans le code grâce à un dossier commun sur Github, et nous avions ajouté un fichier variables.js au .gitignore pour stocker tout ce qui était propre à nos environnements respectifs, sans que cela soit partagé sur la partie commune.

Écrans principaux :

Écran d'accueil :

L'écran d'accueil est chargé d'appeler une première page de 36 résultats, et d'en charger 36 supplémentaires à chaque fois que l'utilisateur parvient en bas du scroll, à condition bien sûr qu'il n'ait pas encore chargé la totalité des requêtes correspondantes à ses filtres. Si l'écran d'accueil n'a pas de paramètre de navigation "foryou : true", alors il charge les requêtes correspondantes à son mot-clé de recherche, et à son filtre "NSFW" (par défaut "false", à moins qu'un utilisateur connecté le mette en "true"). Si l'écran d'accueil a un paramètre de navigation "foryou : true", alors il charge les requêtes correspondantes aux jeux les plus mis en favoris par l'utilisateur.

Écran de détails :

L'écran des détails présentera davantage de précisions que les cartes de l'infinite-scroll.

-Il charge dans un premier temps, les tags en anglais présents dans notre base de données, puis il ira scraper les tags et les reviews correspondants à la langue, anglaise si aucun utilisateur de connecté, ou que l'utilisateur a renseigné dans ses préférences si son compte est connecté.

-Dans la première requête, une description détaillée et des médias sont aussi chargés. Les médias sont les vidéos ou les images du jeux, et sont navigables via le lecteur de média en caroussel.

-Si un utilisateur est connecté, il a la possibilité d'ajouter ou retirer le jeux dans ses favoris.

Composants principaux :

  • Header
  • Carte d'un jeu
  • Lecteur Vidéo/Images Caroussel
  • Avatar personnalisé
  • Boutons du profil
Header :

Le header est le composant le plus réutilisé, il est présent sur la plupart des pages, permet de revenir en arrière dans la navigation, d'ouvrir le menu de navigation (ne sera réellement utile que pour les évolutions futures), d'accéder à son profil si connecté, ou à l'écran de connexion sinon. Il permet aussi de rechercher un jeu dans la base de données, la valeur de la recherche est stockée dans les données de l'application, grâce à redux, pour une UX plus agréable à travers les navigations.

Carte d'un jeu :

La carte d'un jeu, permet d'accéder à la page de détail du jeu sur lequel on clique, elle possède également une version extensible en hauteur pour afficher davantage de détails sans naviguer ailleurs.

Avatar personnalisé et boutons du profil :

-L'avatar personnalisé est utilisé en grande échelle sur l'écran du profil, et en petite icône sur le header, sa logique est la même pour les deux : si l'utilisateur a renseigné une photo d'avatar, alors elle est utilisée, sinon un avatar personnalisé est créé avec les deux premières lettres de l'username de l'utilisateur.

-Les boutons du profil sont unifiés dans leurs UI, on viendra simplement leur confier des logiques, textes et icônes différentes.

Gestion de la responsivité

Responsivité UX

Au niveau de l'UX, l'utilisateur reçoit des informations sur les processus sous-jacents grâce à plusieurs moyens :

  • Dépendance Toast : Des hooks personnalisés sont mis en place pour lancer des popups temporaires (toasts) informant de la réussite ou de l'échec d'une action.
  • Indicateurs d'activités : Des loaders personnalisés sont implémentés.
  • Props "loading" de la dépendance UI react-native-paper : Les composants "activables" peuvent être rendus inactifs en même temps que montrer qu'ils sont en train de traiter une action.
Responsivité UI
Au niveau de l'UI, l'application gère les différentes tailles d'écrans grâce à 3 moyens :
  • Flexbox CSS : Utilisation des propriétés flex pour l'adaptation dynamique
  • Dimensions relatives : Propriétés de taille relative (ex: height="25%")
  • Dimensions React Native : Import et utilisation de l'API Dimensions pour les calculs adaptatifs

Les cas les plus simples sont gérés via les deux premières solutions, mais le gros de la responsivité est géré grâce à la troisième.

En demandant à une IA de faire l'agrégation des ratios hauteur/largeur des écrans de tablette Android pour trouver le minimum (entendre "la tablette la plus proche d'un carré"), on s'approche d'un ratio 4/3, donc comme notre application est verrouillée en mode portrait, et que les écrans sont dans des ScrollViews pour gérer les débordements verticaux (worst case scenario ratio hauteur/largeur trop élévé, géré), si on affiche correctement nos composants proportionnellement à la largeur d'écran, sur un format 4/3, on aura un affichage correct de partout (worst case scenario ratio hauteur/largeur trop bas, géré) CQFD.

Le composant ayant le ratio le plus "étiré en hauteur" est la carte d'un jeu de l'infinite-scroll lorsqu'on a cliqué sur "voir plus", on l'affiche correctement sur du 4/3, l'utilisateur voit les détails qu'il est censé voir sans avoir besoin de scroller verticalement, donc il n'y aura aucun problème de responsivité nulle part si on affiche tous nos composants de la même manière : proportionnellement à la largeur de l'écran.

Sécurisations des routes :

Chaque route est déclarée avec son endpoint et son contrôleur appelé, et les middlewares d'authentification nécessaires pour y accéder. En exemple je mets le cas le plus complexe de ma seule route nécessitant le rôle d'admin, celle pour ajouter un nouveau jeu au catalogue :

isAuthenticated vérifie la présence d'un cookie appelé "token" puis le décrypte en tant que JSON Web Token en utilisant le même secret qu'à l'encodage, secret stocké dans un fichier config.env et accessible grâce à la dépendance dotenv. Si une erreur se produit (pas de token, token invalide, ou token renvoyant à un utilisateur inexistant) le middleware d'erreur est immédiatement appelé, sautant ainsi les étapes intermédaires, notamment l'appel du contrôleur. Si aucune erreur ne s'est produite, on renseigne l'utilisateur à qui appartient le token dans la requête, avant de la passer ici, au middleware d'authentification suivant, sinon au contrôleur.

Ici isAdmin vérifie simplement que l'utilisateur qui a validé son token via isAuthenticated, possède bien le rôle d'admin, et soulève une erreur pour ne pas appeler le contrôleur si ce n'est pas le cas.

Composants de connexions

Composant de création de compte :

On récupère les informations nécessaires à la création d'un nouvel utilisateur, on s'assure que son e-mail ne soit pas déjà dans notre base de données, et si tout est valide, on crée l'utilisateur, et on renvoie un token en réponse.

Composant de connexion :

On trouve l'utilisateur correspondant à l'e-mail fourni en corps de requête, s'il est bien trouvé, on compare le mot de passe fourni, avec celui en base de données, si tout est valide, on renvoie un token en réponse.

Composant d'envoi de token :

On génère un token correspondant à l'utilisateur passé en paramètre, et aux règlages du mode dans lequel le serveur a été lancé, développement ou production, puis on le renvoie à l'utilisateur dans un cookie.

Méthodes du modèle User :

La méthode save est appelée à chaque fois qu'on enregistre une modification sur un utilisateur, que ce soit lors d'une création ou d'une mise à jour. On définit donc une méthode qui sera appelée avant chaque appel de "save", et on s'assure que si l'utilisateur ne souhaite pas modifier son mot de passe, on ne le réencrypte pas une nouvelle fois. Toutefois, si le mot de passe est modifié, ou initialisé, on le stocke encrypté en base de données, grâce à la dépendance bcrypt.

On définit une fonction qui sera appelée sur un utilisateur, et qui compare son mot de passe à celui passé en paramètre, toujours avec la dépendance bcrypt.

La méthode pour générer un token à l'utilisateur encodera dans le JWT le minimum d'informations pour le retrouver, à savoir son identifiant MongoDB. On le signe bien avec le secret présent dans config.env, et avec une durée d'expiration arbitraire de 15 jours, puis on le retourne.

Composant de recherche

L'utilisateur cherche dans la liste de jeux selon un mot-clé qui sera contenu dans le titre, et filtre les jeux NSFW en fonction de ce qu'il a renseigné dans ses préférences utilisateur.

Il parcourra la liste de jeux obtenus, grâce à un scroll sans fin (infinite-scroll), qui chargera les résultats de la liste 36 par 36 (divisibilité exceptionnelle, pour une future évolution en affichage grid, responsive selon le niveau de zoom, pour navigateurs web).

On renvoie également la taille totale de la recherche, pour que le frontend puisse savoir qu'il a atteint la fin de la liste, et ne demande pas à charger davantage de résultats.

Composant de recommandation

Complexité logicielle : O(1) si la liste de favoris n'a pas été modifiée, sinon calcul des 5 tags les plus présents en O(n*log(n)).

Je souhaite avoir une fonction prédictive, donc pour que les résultats ne dépendent pas d'une forme de hasard, l'algorithme calcule les 5 tags les plus présents, et retourne ces 5 tags, concaténés à tous ceux qui ont exactement la même fréquence d'apparition que le 5ème tag.

La complexité en O(n*log(n)) provient de l'utilisation d'un algorithme de tri, que l'on sait ne pas pouvoir être plus efficient que cela. En effet, les algorithmes de tri mis à disposition nativement par les langages informatiques, quels qu'il soient, utiliseront tous une approche de type "divide and conquer" pour pouvoir baisser la complexité de O(n²) (quadratique) à O(n*log(n)) (quasi-linéaire), car si c'était plus complexe que cela, le langage serait frauduleux, et si c'était moins complexe, l'exploit que cela constituerait en terme d'algorithmie, tournerait une page dans l'histoire de l'informatique telle qu'on la connaît!

La méthode de tri différe alors de la simple recherche, cette fois on se contentera de retourner les jeux qui ont au moins un tag en commun avec nos tops tags, en revanche la méthode de parcours de l'utilisateur reste la même, infinite-scroll par pagination, on retournera donc aussi la taille de la liste triée.

Composant de webscraping légal

Note légale : Le webscraping est utilisé uniquement dans le cadre d'un projet scolaire, à des fins éducatives et non commerciales.

La dépendance Cheerio est utilisée, pour transformer le texte HTML de la page, en éléments jQuery, donc accessibles via les sélecteurs classiques. En lisant le contenu d'une page de détails d'un jeu sur steam, on repère les combinaisons de sélecteurs nous permettant d'accéder aux balises qui nous intéressent, et uniquement elles, et on s'en sert ensuite pour extraire les renseignements qu'elles contiennent, que l'on retournera à l'utilisateur, formatés selon nos désirs.

CP6 - Définir l'architecture logicielle d'une application

Architecture multi-couches

Backend : Node.js + Express + Architecture MVC avec Mongoose

Base de données : MongoDB Atlas

Services externes : Steam Scraping + Cloudinary

Déploiement : Render (Backend)

Architecture frontend

Technologie principale : React Native avec Expo

Architecture : "MVVM avec Redux" ou "Redux Architecture Pattern"

Persistence des données globales d'application : Redux toolkit

Persistence des données utilisateur : Cookies

Communication avec le backend : Axios

Navigation : React navigation

Librairie UI : React native paper

Gestionnaire fichiers : Dépendances : Expo camera, Expo image picker, et Mime

Tests : Jest

Point d'entrée de l'application : index.js, simplement chargé d'appeler App.js en tant que racine de l'application Expo.

App.js est simplement chargé de wrapper Main.jsx autour du provider et store redux.

Main.jsx a alors deux responsabilités : Charger l'utilisateur connecté dans redux grâce au token présent dans un cookie, puis wrapper les différents écrans de notre application dans un StackNavigator.

Une bonne partie de la sécurité de l'application a déjà été expliquée dans le CP3, mon dossier projet, tout comme mon projet suit au maximum le principe DRY (Don't Repeat Yourself), je me contenterai donc d'expliquer ce qui ne l'a pas déjà été.

Point d'entrée de l'application : server.js, chargé d'établir les connections aux données, l'accés aux variables d'environnements en important app.js, la base de données NoSQL et les images stockées dans Cloudinary. Une fois ces prérequis faits, server.js lance app.js en écoute pour des requêtes.

Le fichier app.js décrit l'ordre d'éxecution des différents processus du serveur express : on rend les requêtes/réponses utilisables en format JSON, on parse les cookies pour les rendre utilisables aussi, on autorise les CORS pour qu'un frontend puisse interagir avec le backend depuis une origine différente en utilisant un système d'authentification.

Par la suite, on configure notre Swagger, pour aisémment documenter notre API, et enfin on définit le pipeline de notre logique d'utilisation de l'API en déclarant l'utilisation de nos routes, avec un routeur pour chaque modèle, suivi de l'utilisation du middleware d'erreur.

Les routeurs déclarent quels endpoints appellent quels contrôleurs, selon quel verbe HTTP, et quels middlewares doivent être appelés avant le controlleur. Il y a 3 middlewares susceptibles d'etre appelés :

-Le middleware d'authentification pour les routes nécessitant d'être un utilisateur connecté.

-Le middleware administrateur qui se mettra toujours à la suite de celui d'authentification, pour ensuite s'assurer que l'utilisateur connecté, possède aussi un rôle "admin".

-Et enfin, le middleware d'upload de fichier, qui va temporairement allouer de la mémoire sur le serveur, le temps de manipuler le fichier envoyé (via la dépendance Multer) et le rendre utilisable dans la requête fournie au contrôleur par la création de sa propriété "file".

Le dernier middleware, celui d'erreur, appelé après le contrôleur en cas d'erreur émises avant ou pendant l'appel du contrôleur, permet quant à lui, de gérer toutes les erreurs au même endroit, en une seule déclaration.

Par exemple, la ligne qui appelle "User.findById(req.user._id)" sera la même pour tous les contrôleurs qui nécessitent d'être authentifié, si suite à cela on ne trouve pas d'utilisateur, on émettra manuellement la même erreur, qui sera traîtée de la même manière, par une portion unique de notre code : dans notre middleware d'erreur. DRY.

De la même manière, lorsque Mongoose, chargé de la validation des données, émettra (naturellement cette fois) une erreur lors d'un "User.create()", ou d'un "user.save()" par exemple, en cas de nom d'utilisateur déjà présent dans la base de donnée, alors que la propriété "username" a été déclarée "unique" dans le schema User, quel que soit le contrôleur qui l'a émise, ce sera la même erreur, et elle sera traîtée au même endroit. DRY.

L'erreur par défaut de la classe Error ne permet pas de transmettre un code d'erreur personnalisé, qu'à cela ne tienne, on déclare une classe fille de Error : ErrorHandler, qui reprend tous les attributs de sa mère, en y ajoutant ce code désiré.

Et pour terminer l'explication sur la gestion des erreurs, voici la fonction "wrapper" qui enveloppera toute notre logique asynchrone, et transmettra au middleware d'erreur, les erreurs avec leurs messages et leurs codes si elles ont été lancées avec notre classe fille, ou simplement leur message si elles ont été émises naturellement.

Cette fonction, évite les répétitions et les nestings illisibles de bloc try-catch, ainsi que tout oubli d'une portion de code qui n'aurait pas été mise dans un bloc try, et pourrait causer le crash du serveur.

CP10 - Préparer et documenter le déploiement d’une application

Deploiement

Pour déployer sur render, j'ai d'abord dû déployer ma base de données en ligne, je suis donc passé sur MongoDB Atlas et son offre gratuite, en limitant mon dataset de jeux.

L'offre gratuite de render est relativement inadaptée pour la mise en production. En effet cela coupe le serveur en cas d'inactivité, qui mettra environ 50 secondes à se relancer à la première requête faite, mais aussi attribue des adresses IP dynamiques à notre backend. J'ai donc dû autoriser toutes les adresses IP à pouvoir se connecter à ma base de données en ligne, ce qui est une faille de sécurité à absolument éviter en cas de mise en production.

Hormis cela, le déploiement est extrêmement fiable et rapide, et se refait automatiquement à chaque modification du repository (ayant validé le pipeline de tests).

J'ai pour le moment, gardé exactement les mêmes variables d'environnement que dans le config.env, même pour les paramètres SMTP : j'ai gardé Mailtrap. Je modifierai cela avec de vrais mails lorsque je déploierai de manière sécurisée avec une IP fixe. J'ai simplement lancé le serveur en mode production, et pas développement comme je le faisais en local. Il a donc fallu rajouter la variable d'environnement NODE_ENV qui était jusque là indépendante du config.env, et était initialisée en fonction du script choisi pour lancer le serveur dans sa version locale.
